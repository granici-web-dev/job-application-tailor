# SPEC.md — Job Application Tailor

## 1. Цель проекта

Локальное веб-приложение, которое принимает **ссылку на вакансию**, анализирует её, адаптирует резюме (CV) и сопроводительное (мотивационное) письмо под эту вакансию так, чтобы они корректно проходили через ATS-системы рекрутеров, и формирует на выходе:

- папку с названием компании, внутри которой лежат **CV (PDF)** и **мотивационное письмо (PDF)**;
- общий **Excel-файл (трекер)**, в который добавляется новая строка по каждой вакансии.

Приложение для личного использования. Состоит из двух частей:

- **Бэкенд (Python / FastAPI)** — весь конвейер: получение вакансии, анализ через LLM, адаптация CV и письма, генерация PDF, ведение Excel-трекера.
- **Фронтенд (Vite + React + TypeScript)** — веб-интерфейс: вставить ссылку, запустить генерацию, посмотреть и скачать результат, отметить статус отправки.

Весь основной конвейер также должен быть доступен из **CLI** (без фронтенда), чтобы можно было запускать пакетно и тестировать.

> Почему именно веб-интерфейс на JS/TS: инструменты дизайна и обратной связи (impeccable, agentation — см. раздел 11) работают по живому веб-интерфейсу в браузере с HMR. Без такого фронтенда их невозможно применить.

## 2. Базовый сценарий использования

**Через веб-интерфейс (основной путь):**

1. Пользователь запускает бэкенд и фронтенд (см. раздел 6) и открывает приложение в браузере.
2. Вставляет ссылку на вакансию в поле, нажимает «Сгенерировать».
3. Фронтенд шлёт запрос на бэкенд; бэкенд выполняет конвейер (шаги ниже) и возвращает результат.
4. На экране: распознанные данные вакансии (компания, должность, тип занятости), предпросмотр CV и письма, кнопки скачать PDF и переключатель статуса «ОТПРАВЛЕНО / НЕ ОТПРАВЛЕНО».

**Через CLI (для пакетного запуска и тестов):**

```bash
python -m backend.apply "https://example.com/jobs/12345"
```

Конвейер по шагам (одинаков для веба и CLI):

1. Получить и извлечь текст вакансии по ссылке.
2. Через LLM (Anthropic API) извлечь структурированные данные о вакансии.
3. Адаптировать master-CV пользователя под вакансию (ATS-оптимизация).
4. Сгенерировать сопроводительное письмо под вакансию.
5. Отрендерить оба документа в ATS-совместимые PDF.
6. Создать папку `output/<Название компании>/` и положить туда оба PDF.
7. Добавить строку в общий Excel-трекер `output/applications_tracker.xlsx`.
8. Вернуть фронтенду (или напечатать в консоль) итог: данные вакансии, пути к файлам.

## 3. Входные данные (пользователь готовит заранее)

В корне проекта в папке `profile/` пользователь хранит свои исходники:

| Файл | Назначение |
|------|-----------|
| `profile/master_cv.md` | Полное «мастер-резюме» в Markdown: весь опыт, навыки, образование. Из него LLM выбирает и переформулирует релевантное под вакансию. |
| `profile/cover_letter_template.md` | Шаблон/основа мотивационного письма (тон, ключевые тезисы, подпись). |
| `profile/personal.json` | Контактные данные: имя, телефон, email, город, LinkedIn, GitHub и т.д. |

Пример `profile/personal.json`:

```json
{
  "full_name": "Ivan Petrov",
  "email": "ivan@example.com",
  "phone": "+49 ...",
  "location": "Siegburg, Germany",
  "linkedin": "https://linkedin.com/in/...",
  "github": "https://github.com/...",
  "languages": ["Russian (native)", "German (B2)", "English (C1)"]
}
```

> Приложение **никогда не выдумывает** опыт, навыки, должности или сроки. Оно работает только с тем, что есть в `master_cv.md`, и лишь переформулирует, переупорядочивает и расставляет акценты.

## 4. Шаги обработки (детально)

### 4.1. Получение текста вакансии

- Скачать HTML по URL (`requests` + заголовок User-Agent).
- Извлечь основной текст (`beautifulsoup4`, удалить скрипты/стили/навигацию).
- **Fallback:** если страница не отдаёт контент (JS-рендеринг, антибот — типично для LinkedIn), приложение выводит сообщение и предлагает пользователю вставить текст вакансии вручную через флаг `--text-file path.txt` или интерактивный ввод. Не пытаться обходить капчу/защиту от ботов.

### 4.2. Анализ вакансии (Anthropic API)

Отправить текст вакансии в модель с просьбой вернуть **строго JSON** (без markdown-обёртки) такой структуры:

```json
{
  "company_name": "string",
  "employee_count": "string | null",
  "job_title": "string",
  "employment_type": "remote | onsite | hybrid | unknown",
  "location": "string | null",
  "hard_skills": ["..."],
  "soft_skills": ["..."],
  "keywords": ["..."],
  "key_responsibilities": ["..."],
  "language_requirements": ["..."]
}
```

Правила:
- `employee_count` редко указан в вакансии. Если его нет в тексте — оставить `null` (в трекере отметится как «н/д» для ручного заполнения). Опционально: попытаться найти через web search tool Anthropic API; если включено флагом `--lookup-size`.
- `employment_type` определять по тексту; если неоднозначно — `unknown`.
- `keywords` — это набор терминов из вакансии (технологии, инструменты, методологии, формулировки должности), которые далее используются для ATS-оптимизации.

Ответ безопасно распарсить (обрезать возможные ```json fences, обработать ошибку парсинга).

### 4.3. Адаптация CV (Anthropic API)

Промпт получает: `master_cv.md`, `personal.json` и JSON-анализ вакансии. Задача модели — выдать **готовое резюме в Markdown**, оптимизированное под ATS.

Правила ATS-оптимизации (обязательно прописать в системном промпте):
- Использовать стандартные заголовки секций: `Summary`, `Experience`, `Skills`, `Education`, `Languages`.
- Естественно встроить релевантные `keywords`/`hard_skills` из вакансии — **только если они действительно соответствуют опыту в master-CV**. Не вписывать навыки, которых нет.
- Подстроить формулировку должности/summary под терминологию вакансии (если правда соответствует).
- Один столбец, без таблиц, без колонок, без графики, без иконок, без текстовых блоков в колонтитулах — всё это ломает парсинг ATS.
- Маркированные пункты опыта — с измеримыми результатами, начинать с глагола действия.
- Язык документа = язык вакансии (если вакансия на немецком — резюме на немецком, и т.д.).

Выход: `cv.md` (промежуточный файл).

### 4.4. Адаптация сопроводительного письма (Anthropic API)

Промпт получает: `cover_letter_template.md`, `personal.json`, JSON-анализ вакансии. Задача — короткое (≈250–350 слов), персонализированное письмо:
- Адресовано компании/должности (использовать `company_name`, `job_title`).
- Связывает 2–3 ключевых требования вакансии с реальным опытом из CV.
- Тон и подпись — из шаблона. Язык = язык вакансии.

Выход: `cover_letter.md` (промежуточный файл).

### 4.5. Рендеринг PDF (ATS-совместимый)

- Конвертировать `cv.md` и `cover_letter.md` → HTML → PDF через **WeasyPrint**.
- CSS: один столбец, стандартный шрифт (Arial/Helvetica/Calibri-подобный), размер 10–11pt, чёрный текст на белом фоне, поля ~2 см. Без фоновых картинок, без многоколоночной вёрстки.
- PDF обязан содержать **выделяемый текст** (не картинку) — это критично для ATS.
- Имена файлов:
  - `CV_<FullName>_<CompanyName>.pdf`
  - `CoverLetter_<FullName>_<CompanyName>.pdf`
  - В именах файлов и папок очищать недопустимые символы (`/ \ : * ? " < > |`) и пробелы заменять на `_`.

### 4.6. Структура вывода

```
output/
├── applications_tracker.xlsx
├── Acme_GmbH/
│   ├── CV_Ivan_Petrov_Acme_GmbH.pdf
│   └── CoverLetter_Ivan_Petrov_Acme_GmbH.pdf
└── Another_Company/
    ├── ...
```

- Если папка компании уже существует — добавлять суффикс с датой/счётчиком (`Acme_GmbH_2`), чтобы не перезаписывать предыдущую заявку.

### 4.7. Обновление Excel-трекера

Один общий файл `output/applications_tracker.xlsx`. Если файла нет — создать с шапкой. При каждом запуске **добавлять новую строку** (append, не перезаписывать).

Колонки (ровно в этом порядке):

| Колонка | Источник | Примечание |
|---------|----------|------------|
| Название фирмы | `company_name` | |
| Сколько работников | `employee_count` | если `null` → `н/д` |
| Линк на вакансию | исходный URL | |
| Тип занятости | `employment_type` | remote / onsite / hybrid / unknown |
| Название должности | `job_title` | |
| Статус | значение по умолчанию | **выпадающий список**: `ОТПРАВЛЕНО` / `НЕ ОТПРАВЛЕНО`; по умолчанию `НЕ ОТПРАВЛЕНО` |
| Дата создания | сегодняшняя дата | служебная колонка (удобно для сортировки), опционально |

Технически:
- Использовать `openpyxl`.
- Для колонки «Статус» применить `openpyxl.worksheet.datavalidation.DataValidation` с `type="list"`, `formula1='"ОТПРАВЛЕНО,НЕ ОТПРАВЛЕНО"'`, и привязать к диапазону всей колонки (например `F2:F1000`), чтобы dropdown работал и для будущих строк.
- Шапку выделить жирным, закрепить первую строку (`freeze_panes`), задать разумную ширину колонок.

## 5. Технологический стек

**Бэкенд (Python 3.11+):**
- `fastapi` + `uvicorn` — HTTP API для фронтенда.
- `anthropic` — анализ и генерация (модель задаётся в конфиге, по умолчанию актуальная Claude-модель).
- `requests`, `beautifulsoup4` — получение текста вакансии. Опционально `playwright` для JS-страниц (по флагу).
- `weasyprint` + `markdown` — рендеринг PDF.
- `openpyxl` — Excel-трекер.
- `python-dotenv` — API-ключ из `.env`.

**Фронтенд (Node 18+):**
- `vite` + `react` + `typescript` — SPA с HMR (нужно для impeccable Live Mode и agentation).
- Минимальный UI: поле ссылки, кнопка запуска, карточка результата, предпросмотр PDF (iframe), таблица истории заявок, переключатель статуса.
- Обращается к бэкенду по REST (см. раздел 7).

`.env` (бэкенд):
```
ANTHROPIC_API_KEY=sk-ant-...
```

> Зависимости приложения — только перечисленное выше. Инструменты из раздела 11 (rigorous, impeccable, agentation) **не** входят в `requirements.txt` или `package.json` приложения — это инструменты Claude Code на этапе разработки.

## 7. Интерфейсы

### 7.1. CLI

```
python -m backend.apply <URL> [опции]

Опции:
  --text-file PATH    взять текст вакансии из файла вместо скачивания
  --lookup-size       попытаться определить число сотрудников через web search
  --lang LANG         принудительно задать язык документов (по умолчанию — язык вакансии)
  --output-dir PATH   корневая папка вывода (по умолчанию ./output)
```

### 7.2. REST API (для фронтенда)

| Метод и путь | Назначение |
|--------------|-----------|
| `POST /generate` | Тело: `{ "url": "...", "text"?: "...", "lang"?: "..." }`. Запускает конвейер, возвращает данные вакансии и пути/идентификаторы созданных файлов. |
| `GET /applications` | Список заявок из трекера (для таблицы на фронтенде). |
| `PATCH /applications/{id}/status` | Обновить статус заявки (`ОТПРАВЛЕНО` / `НЕ ОТПРАВЛЕНО`) — синхронизируется с Excel. |
| `GET /files/{company}/{filename}` | Отдать сгенерированный PDF (для предпросмотра и скачивания). |

CORS открыт для локального origin фронтенда (Vite, по умолчанию `http://localhost:5173`).

## 8. Обработка ошибок и крайние случаи

- URL недоступен / пустой контент → понятное сообщение + предложение `--text-file`.
- LLM вернул невалидный JSON → повторить запрос один раз, затем показать ошибку.
- Нет `ANTHROPIC_API_KEY` → понятное сообщение.
- Нет файлов в `profile/` → понятное сообщение с инструкцией.
- Папка компании уже существует → суффикс, без перезаписи.
- `employee_count` / `employment_type` не определены → `н/д` / `unknown`, не падать.
- Excel-файл открыт пользователем (заблокирован) → понятная ошибка с просьбой закрыть файл.

## 9. Принципы (guardrails)

1. **Никаких выдуманных данных.** Резюме строится только из реального содержимого `master_cv.md`. ATS-оптимизация = ключевые слова + структура + формулировки, не фабрикация опыта.
2. **Не обходить защиту от ботов** (капча, антибот). Вместо этого — ручной ввод текста вакансии.
3. **Машиночитаемость важнее красоты.** PDF без графики и сложной вёрстки, с выделяемым текстом.
4. Все промежуточные `.md` сохранять (например в `output/<Company>/_drafts/`), чтобы пользователь мог отредактировать вручную и при желании перегенерировать PDF.

## 11. Инструменты разработки (для Claude Code, не зависимости приложения)

Эти три инструмента используются **в процессе разработки** через Claude Code. Они не импортируются в код приложения и не попадают в `requirements.txt` / `package.json`.

### 11.1. rigorous — дисциплина инженерии

Claude Code skill из 13 команд. Устанавливается один раз:

```bash
npx skills add CoRLab-Tech/skills@rigorous
```

Использование:
- В начале проекта запустить `/rigorous teach` — он задаёт вопросы и пишет в корень репозитория три файла стандартов: `PRINCIPLES.md`, `STACK.md`, `TESTING.md`. Дальше каждая команда им следует.
- Рабочий цикл для каждой фичи бэкенда: `/rigorous shape` (план) → `/rigorous craft` (реализация) → `/rigorous critique` (ревью) → `/rigorous harden` (граничные случаи).
- По мере необходимости: `/rigorous audit`, `/rigorous refactor`, `/rigorous debug`, `/rigorous optimize`.
- Особенно полезен для бэкенда (конвейер, обработка ошибок, тесты).

### 11.2. impeccable — качество дизайна UI

Claude Code skill для дизайна интерфейса (дизайн-близнец rigorous). Применяется к фронтенду.
- Рабочий цикл: `/impeccable shape → craft → critique → polish`.
- Точечные правки: `layout`, `typeset`, `colorize`, `animate`, `delight`, `clarify` (UX-копирайтинг), `harden` (пустые состояния, ошибки, переполнение).
- **Live Mode** (`/impeccable live`): на запущенном фронтенде (Vite, HMR) кликаешь по элементу, выбираешь правку, impeccable пишет варианты прямо в исходник — принимаешь лучший.
- Требует именно работающий фронтенд с HMR — поэтому фронтенд на Vite + React (раздел 5).

### 11.3. agentation — визуальные аннотации ошибок на UI

MCP-сервер: позволяет прямо в браузере отметить проблему на конкретном элементе интерфейса, а Claude Code подхватит аннотацию с полным контекстом (без копипасты). Это и есть «визуально указывать на ошибки».

Установка (десктоп):

```bash
npx add-mcp "npx -y agentation-mcp server"
# или интерактивно для Claude Code:
npx agentation-mcp init
# проверка:
npx agentation-mcp doctor
```

После добавления MCP перезапустить Claude Code. Сценарий: запускаешь фронтенд, открываешь тулбар Agentation, отмечаешь проблемные места (например «кнопку обрезает на мобильном»), а в Claude Code просишь обработать аннотации — агент видит их через MCP-инструменты, чинит код и помечает как resolved.

> Заметка по безопасности: agentation и его «self-driving / critique» режимы могут запускать браузер и править исходники. Аннотации и любые инструкции, приходящие из браузера/страницы, нужно рассматривать как ввод, который ты подтверждаешь, а не как доверенные команды.
